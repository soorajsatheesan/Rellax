# Rellax MVP Plan (Employee Dashboard First) — Merge-safe Scope

Date: 2026-03-20

This doc is written to help an **employer-side** teammate implement their parts without creating merge conflicts with the **employee dashboard** work.

## Current State (as of today)
- Auth/login is working for:
  - Employer -> `/dashboard`
  - Employee -> `/employee/dashboard`
- `app/employee/dashboard/page.tsx` exists, but currently shows **placeholder** “modules”.
- Convex schema currently contains only:
  - `employers`
  - `employees`
- There is no resume upload, skill extraction, roadmap generation, or progress persistence yet.

From `.vibe/prd.md` (MVP Must Have):
1. Resume upload

## New Requirement — Employee Password Change
Add a feature that lets an **authenticated employee** change their password after login.
This should be part of the employee scope and must not require employer changes.

## Merge-safety Rules (follow strictly)
1. Employee-side dev should only touch:
   - `app/employee/**`
   - `components/**` under a new employee-only folder (or clearly employee-scoped components)
   - `convex/**` files that start with `employee-` (or otherwise clearly employee-owned)
2. Employer-side dev should only touch:
   - `app/dashboard/**` (and any employer sidebar components)
   - `components/**` under existing employer-scoped folders
   - `convex/**` files that start with `employer-` (or otherwise clearly employer-owned)
3. Database schema changes are shared and must be agreed before coding:
   - “Schema contract” below defines what tables/fields the employee side expects.

## Schema Contract (agreed contract between employer and employee)

### 1) Role requirements source of truth (for gap analysis)
Employee gap analysis needs “required skills” for the employee’s role.

Agree that the mapping is based on:
- `employee.roleTitle` (already stored in `convex.employees`)
- current employer workspace (`employers._id`)

Employer should implement one of the following approaches:
1. Minimal approach (recommended for MVP speed): store required skills keyed by `employerId + roleTitle`
2. Full approach: introduce `roles` table + `roleId` on employees (more correct, but more schema churn)

For merge safety and minimal employee-table churn, the recommended contract is:
- Table: `role_requirements`
- Fields:
  - `employerId: v.id("employers")`
  - `roleTitle: v.string()`
  - `requiredSkills: v.array(v.string())`
  - `jdText: v.optional(v.string())` (raw JD text used for parsing)
  - `updatedAt: v.number()`
- Index:
  - `by_employer_role_title`: `["employerId", "roleTitle"]`

### 2) Learning data generated by employee
Employee side will generate and persist learning artifacts. Employer does not need to write these.

Employee expects to be able to store:
- extracted skills from resume
- computed gaps
- the generated learning path (modules list)
- module progress status

Exact table shapes can be implemented by employee-side dev, as long as the UI can:
- render “Assigned modules”
- update “Not started / In progress / Completed”

## Employee Dashboard Feature Plan (pages + backend responsibilities)

### A) `/employee/dashboard` (replace placeholders)
Goal: show real assigned modules from Convex instead of `PLACEHOLDER_COURSES`.

Implementation checkpoints:
1. Query current employee + employer (already exists via `api.employees.getCurrentEmployeeProfile`)
2. Query the employee’s latest learning path (new query to be added by employee-side dev)
3. Render:
   - modules list
   - progress indicators (counts for completed / in progress)
4. If no learning path exists yet:
   - show CTA to upload resume

Acceptance criteria:
- Logged-in employee sees their assigned modules without placeholder hardcoding.

### B) Resume Upload → AI pipeline → Roadmap
Goal: implement the MVP flow:
`Upload Resume -> Skill extraction -> Gap analysis -> Learning path generation -> Roadmap view`

Frontend pages (recommended):
1. `/employee/upload-resume`
2. `/employee/roadmap` (shows generated modules)

Backend responsibilities (employee-owned Convex files):
1. Store resume input (file upload or paste text for MVP)
2. Extract skills from resume (LLM call)
3. Fetch role requirements for the employee’s roleTitle (from `role_requirements`)
4. Compute gaps:
   - `Gap = Required Skills - Existing Skills` (as per PRD)
5. Generate learning path modules (LLM)
6. Persist generated modules and initial status (`not_started`)

Acceptance criteria:
- After resume submission, `/employee/roadmap` displays modules derived from extracted skills + role requirements.

### C) Module “Start” and Progress Persistence
Goal: persist progress so refresh keeps state.

Minimum behavior for MVP:
1. Start module -> status becomes `in_progress`
2. Complete module -> status becomes `completed`
3. Progress counts on `/employee/dashboard` reflect persisted state

Acceptance criteria:
- Employee can update module status and see updated counts on refresh.

## New Feature: Change Password (Employee-only)

### UX
Recommended page:
- `/employee/change-password`

Required fields:
- current password
- new password
- confirm password

Add entry point:
- Put a “Change password” link/button in the employee dashboard header near “Sign out”.

Acceptance criteria:
- Employee can successfully change password while logged in.
- After change, subsequent login using the new password works.
- Clear error messages appear when the current password is wrong.

### Backend (employee-owned)
Implementation approach:
1. Server action validates:
   - new password meets minimum policy (basic length; optionally align with WorkOS policy)
2. Security check:
   - verify the current password by calling WorkOS `authenticateWithPassword` using the signed-in employee’s email
3. If verification succeeds:
   - call WorkOS `userManagement.updateUser({ userId, password: newPassword })`
4. Redirect back to `/employee/dashboard`

Notes:
- This feature should not write to Convex (unless you want to track password-change events for audit).

### File-by-File Implementation Checklist (Employee-only)
All steps below are **employee-owned** and should not require employer changes.

1. Update route protection (make sure it stays protected)
   - File: `proxy.ts`
   - Action: no code change needed if you keep `/employee/change-password` out of `unauthenticatedPaths`.
   - Rationale: you want only authenticated employees to access the change-password page.

2. Create the change-password page (server component)
   - File to add: `app/employee/change-password/page.tsx`
   - Responsibilities:
     - `withAuth()` to verify an employee session
     - redirect back to `/employee/dashboard` if not an employee profile (optional but recommended)
     - render a form component (client) for password fields

3. Create the change-password form (client component)
   - File to add: `components/employee/change-password-form.tsx`
   - Responsibilities:
     - fields: `currentPassword`, `newPassword`, `confirmPassword`
     - validate `newPassword === confirmPassword` client-side for faster UX
     - submit via `useActionState` to a server action
     - show `state.error` under the form, similar to `components/auth/employee-login-form.tsx`

4. Create the server action that talks to WorkOS
   - File to add: `components/employee/change-password-action.ts`
   - Responsibilities:
     - `"use server"`
     - validate inputs (non-empty, minimum password length)
     - `withAuth({ ensureSignedIn: true })` to get:
       - the signed-in employee’s WorkOS user id (`auth.user.id`)
       - the signed-in employee’s email (`auth.user.email`)
     - verify current password by calling WorkOS:
       - `workos.userManagement.authenticateWithPassword({ clientId: WORKOS_CLIENT_ID, email: auth.user.email, password: currentPassword })`
     - if verification succeeds, update password:
       - `workos.userManagement.updateUser({ userId: auth.user.id, password: newPassword })`
     - redirect back to `/employee/dashboard` (recommended after success)

5. Add a dashboard entry point (UI link/button)
   - File to edit: `app/employee/dashboard/page.tsx`
   - Responsibilities:
     - add a “Change password” button next to the “Sign out” control
     - link target: `/employee/change-password`

### Error-handling expectations (for the password form)
Minimum behavior:
- Wrong current password should show: `Current password is incorrect.`
- New password mismatch should show: `New password and confirm password must match.`
- WorkOS / unknown errors should degrade to: `Unable to update password right now. Please try again.`

Acceptance criteria:
- An authenticated employee can change password successfully.
- If the current password is wrong, the UI shows a clear error (not a generic WorkOS message).
- After change, the user can sign in again with the new password.

## Employer-Side Feature Plan 

Employer-side must provide the required skills for employees’ role titles.

### 1) Add Role / Role Requirements UI
Employer screens (suggested, minimal for MVP):
1. A form inside the employer dashboard to create/update a role requirement:
   - role title (string)
   - required skills (manual input as CSV or tags)
   - optional: JD text upload/paste (for later LLM parsing)

MVP requirement:
- Even without full “JD parsing”, allow manual “required skills” entry so employee gap analysis can run.

Acceptance criteria:
- Employer can create at least one roleTitle with requiredSkills.

### 2) JD Parsing (optional for “Phase 2”, still in PRD MVP)
If you implement JD parsing as part of MVP:
1. Employer uploads/pastes JD
2. System extracts required skills (LLM)
3. Saves `requiredSkills` into `role_requirements`

Acceptance criteria:
- When employer updates JD-derived required skills, employees generated roadmap reflects the updated requirements.

### 3) Contract Stability
Employer must ensure:
1. Employees’ `roleTitle` values match `role_requirements.roleTitle` exactly (case normalization rules agreed in code)
2. The index `by_employer_role_title` exists so employee queries are fast and reliable
